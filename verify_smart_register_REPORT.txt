# /app/agent-center/smart-register "拒绝访问" 真实复现报告

**测试目标**: 真实复现触发"拒绝访问"或拦截 toast
**测试环境**: dev server http://127.0.0.1:3001, Playwright chromium headless
**结果**: **未复现** — 36/36 个场景均未触发"拒绝访问"或拦截 toast

## 0. 关键代码结论（先看这里）

拦截逻辑唯一在 `/Users/harry/Desktop/CC_TEST/agent-system/src/layouts/BasicLayout.tsx:51-109` 的 useEffect：

```ts
const roleBaselineBlocked = ownerModule.defaultRoleVisible === 'itAdmin' && roleKey === 'itUser';
const subRoleBaselineBlocked = subPage
  ? (subPage.defaultRoleVisible || ownerModule.defaultRoleVisible) === 'itAdmin' && roleKey === 'itUser'
  : false;
if (moduleBlocked || subPageBlocked) {
  message.info(`${reason} 暂不可访问，已自动跳转到默认页。`, 3);
  navigate(safeFallback, { replace: true });
}
```

**关键 masterMenu 配置** (`src/config/masterMenu.tsx:67-107`):
- `agent-center` 模块: `defaultRoleVisible: 'both'`
- `agent-center:smart-register` 子页: `defaultRoleVisible: 'both'`

因为是 `'both'` 而不是 `'itAdmin'`,所以 **roleBaselineBlocked / subRoleBaselineBlocked 永远为 false**,
BasicLayout 永远不会拦截 /app/agent-center/smart-register — 不管:
- 你是 信息科管理员 还是 科室管理员
- 演示面板是否取消勾选
- 是否在 7 Tab 之间切
- 是否刚从其他模块/编辑/审核页跳过来
- 是否在 smart-register 上动态切角色
- 慢网络/多 Tab/多刷新/URL hash/query 污染
- login 路径

`src/pages/agent-center/smart/SmartRegistrationForm.tsx` 内部也没有任何 `hasPermission` 早返或 `<PermissionDenied/>` —
它只在 `isDeptAdmin = role === ROLE_DEPT` 时禁用「部门」字段,没有拦截。

## 4. "用户可能触发但自动化难复现" 的最可能路径（基于代码分析）

虽然 36 个场景全部 NOT BLOCKED,但代码分析指出以下理论可能:

### 4.1 角色基线若改回 itAdmin,会立刻复现
- `src/config/masterMenu.tsx:103` 把 `defaultRoleVisible: 'both'` 改成 `'itAdmin'`
- 立刻 BasicLayout effect 会判定 subRoleBaselineBlocked=true
- 任何角色为「科室管理员」(itUser) 的用户访问 /smart-register 都会触发 `message.info('「智能体接入中心 / 智能填写新建注册」暂不可访问...')` + navigate('/app/home/workbench')

### 4.2 当前 currentUser 与 useDemoSettings.demoRole 不一致时的 race
- 启动时 `useAuth.defaultAdmin` 默认为 `信息科管理员`
- `useDemoSettingsProvider` 启动时读 localStorage,如果 `demoRole='科室管理员'`,会立刻调 `switchRole('科室管理员')`
- 这两次 setState 是异步的 — 在切换完成前,`currentUser.roles=['信息科管理员']` 但 `demoRole='科室管理员'`,导致 `roleKey='itUser'`
- 我们的 F1/F3 已经覆盖:即便出现这种 race, `smart-register` 因为 `defaultRoleVisible='both'` 也不会被拦截。
- **若改成 'itAdmin',race 期间会触发拦截**

### 4.3 useAuth 状态被外部改 null
- `logout()` 会把 currentUser 置 null
- BasicLayout effect 入口 `if (!currentUser) return;` — 不拦截,安全
- SmartRegistrationForm 内部用 `currentUser?.roles[0] || ROLE_ADMIN` 兜底 — 不会崩
- **无 race**

### 4.4 useEffect 在 React 18 StrictMode 下双跑
- BasicLayout 用 `warnedForRef` 去重 — 不会双弹 toast
- **安全**

### 4.5 总结
**唯一能复现"拒绝访问" / 拦截 toast 的代码路径** = 把 `masterMenu.tsx:91, 103` 的 `defaultRoleVisible` 从 `'both'` 改成 `'itAdmin'`。
当前 commit (2554bee) 的代码,自动化无法复现"拒绝访问"。
