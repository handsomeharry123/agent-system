# 医小管-智能体建设需求管理场景-需求收集场景设计

## 一、场景定位

| **项** | **说明** |
| --- | --- |
| 场景名称 | 需求登记 |
| 操作完成方向（预设） | 1.用户进入场景即默认开始需求收集（无需用户先发起，由医小管自动打招呼、主动发问）；
2.沿固定任务轨道 N0→N9 收集并确认 8 项必填信息（功能描述、提出原因、诊疗环节、提出科室、所需资源、需求紧急程度、提出人、联系方式），经汇总确认后由系统生成需求（自动生成序号、需求标题≤30字、提出时间）；
3.收尾后仅开放三个动作：
①智能化匹配（自动执行，回填匹配情况并展示最高匹配度）
② 需求文档查看/下载（Word、PDF 版）
③ 查看详情（进入需求详情页） |
| 进入方式 | 1.意图路由：命中「我想提需求／想建一个智能体／帮我登记个需求」等口令；
2.快捷入口：对话页常驻【需求登记】；
3.从需求管理页等相关入口进入。
无论何种方式，进入即由医小管自动打招呼、主动发起 |
| 完成标志 | 1.求生成成功（进入需求管理数据集，列表只显示已生成好的需求）
2.智能化匹配执行完成、匹配情况回填并播报
3.用户获得【需求文档查看/下载】与【查看详情】入口，场景锁释放 |
| 产出物 | 1.一条完整需求记录（含回填的匹配情况）；
2.可查看/下载的需求文档（Word、PDF 版）；
3.匹配度最高的 3 个已纳管智能体推荐结果 |
| 预计步数 / 时长 | 约 8 步、3–5 分钟 |

## 二、任务轨道（状态机节点表）

> 每节点独立一行；`动作`只写要点，通用规则不写；`完成条件`是节点被视为已完成的系统条件（含本节点会触发的工具调用）。
> 

| **节点** | **名称** | **动作（要点）** | **完成条件** |
| --- | --- | --- | --- |
| N0 | 场景进入·自动打招呼 | 固定开场：「你好！我是医小管，我来帮您登记智能体建设需求（约 8 步、3–5 分钟，内容随时可改）。请用一两句话描述：您想建设一个什么样的智能体、主要解决什么问题？」 | 用户已给出首轮需求描述（语音／文字任一，内容非空） |
| N1 | 需求描述采集 | 从用户回答（语音转写／文字）中全量抽取：功能意向、服务对象、输入输出、科室与环节线索、痛点线索，写入槽位草稿 | 已从用户回答中抽取到“工作内容”相关信息（不少于一条有效线索） |
| N2 | 功能描述补全 | 1.对照四要素（工作内容、服务对象、输入信息、输出结果）逐项追问缺项（每轮一问）；
2.整理为 ≤500 字规范文本，发送字段确认卡（可点击／语音／文字确认或修改） | 功能描述已整理为 ≤500 字的规范文本，且用户在功能描述确认卡上完成确认 |
| N3 | 提出原因 | 1.围绕“为什么要建”追问现状、痛点、预期收益；
2.整理为 50–300 字，发送字段确认卡（可点击／语音／文字确认或修改） | 提出原因已整理为 50–300 字的规范文本，且用户在提出原因确认卡上完成确认 |
| N4 | 诊疗环节 | 1.依据功能描述推断环节并请确认；
2.无法推断时发送 9 选 1 选项卡（导诊分诊／预问诊／预约挂号／辅助检查／辅助诊断／辅助治疗／住院／手术／其他）；
3.选“其他”请填写 ≤20 字 | 用户已在 9 项诊疗环节（导诊分诊／预问诊／预约挂号／辅助检查／辅助诊断／辅助治疗／住院／手术／其他）中确认一项；若选“其他”，另附 ≤20 字自定义说明 |
| N5 | 提出科室 | 1.对已抽取科室调用 match_department；
2.未提及则询问；
3.模糊命中发送候选选项卡（可点击／语音／文字确认） | 提出科室已匹配到院内科室字典中的唯一科室，且用户完成确认 |
| N6 | 资源与紧急程度 | 1.询问所需资源（业务系统、模型两部分），用户不明确时给建议值供确认；
2.随后发送需求紧急程度选项卡（高／中／低，可点击／语音／文字选择） | 所需资源（业务系统＋模型）与需求紧急程度（高／中／低）两项均已填写，且用户完成确认 |
| N7 | 提出人与联系方式 | 1.请用户提供姓名（2–10 字）与 11 位手机号，逐项调用 validate_field；
2.说明联系方式仅用于评审沟通、为需求提交必要信息 | 姓名（2–10 字，不含数字与特殊字符）与手机号（11 位数字）均通过字段校验 |
| N8 | 汇总确认 | 1.发送汇总卡：逐项列出全部字段，每项带【修改】，底部【确认提交】；
2.用户可点击【修改】某项、也可说／打字“修改××”→跳转对应节点处理后返回 N8 重新汇总 | 用户在汇总卡上完成“确认提交”操作（点击【确认提交】、或语音／文字回复“提交”，三种方式任一即可） |
| N9 | 生成与匹配 | 1.调用 generate_requirement（系统生成序号、需求标题≤30字并查重、提出时间）；
2.播报需求生成成功卡（附【需求文档查看/下载】Word、PDF）；
3.自动执行 match_agents，播报平台内已纳管的 3 个匹配度最高智能体并回填“匹配情况”（最高匹配度）；
4.收尾语提示【需求文档查看/下载】【查看详情】 | 需求已生成（含序号、需求标题、提出时间），智能化匹配已完成并播报 3 个匹配度最高的智能体，“匹配情况”字段已回填最高匹配度，且已向用户提供【需求文档查看/下载】与【查看详情】入口 |

## 三、字段清单

> 只列本场景相关字段；类型 / 校验优先复用《平台通用规范 v1》第九章基线；差异在「规则」列写明。
> 

| **字段** | **来源** | **类型** | **规则（差异化）** |
| --- | --- | --- | --- |
| 序号 | 系统生成 | 自动编号 | 按需求创建先后自动生成，对话中不涉及 |
| 需求标题 | 系统生成 | 短文本 | 1.生成需求时依据功能描述自动生成，一句话概括，≤30 字；
2.查重避免与已有需求重复，重复时自动补充限定词改写 |
| 提出科室 | 对话获取 | 枚举（院内字典） | 1.必须命中院内科室字典；
2.口语别名模糊匹配给候选 |
| 提出原因 | 对话获取 | 长文本 | 1.业务背景与痛点，说明“为什么要建”，50–300 字；
2.由智能体整理 |
| 提出人 | 对话获取 | 短文本 | 1.2–10 个字，不含数字与特殊字符；
2.支持用户修改 |
| 联系方式 | 对话获取 | 手机号 | 1.格式校验，限制 11 个数字；
2.校验失败固定文案“请输入正确的11位手机号”；
3.支持用户修改 |
| 诊疗环节 | 对话获取 | 单选枚举 | 1.单选：导诊分诊／预问诊／预约挂号／辅助检查／辅助诊断／辅助治疗／住院／手术／其他；
2.“其他”填空 ≤20 字 |
| 功能描述 | 对话获取 | 长文本 | 1.覆盖四要素：工作内容、服务对象、输入信息、输出结果；
2.≤500 字，超限自动压缩；
3.由智能体整理 |
| 所需资源 | 对话获取 | 结构化文本 | 1.业务系统、模型两部分；
2.用户不明确时由智能体给出建议值供确认 |
| 需求紧急程度 | 对话获取 | 单选枚举 | 高／中／低，由提出人建议 |
| 匹配情况 | 动作回填 | 百分比＋列表 | N9 自动执行智能化匹配：推荐平台内已纳管的 3 个匹配度最高智能体（编号、名称、匹配度%），字段展示最高匹配度 |
| 提出时间 | 系统生成 | 日期时间 | 系统自动记录需求提交时间，格式 YYYY-MM-DD HH:MM |
