# 医小管-智能体接入审核场景-需求说明

## 一、场景定位

| **项** | **说明** |
| --- | --- |
| 场景名称 | 接入审核 |
| 使用角色 | 信息科管理员（具备审核权限的平台管理员） |
| 人机分工 | 1.医小管负责"预审"：对每个待审核智能体的注册信息自动核验并给出建议结论与依据（建议通过／建议退回修改）；
2.管理员负责"二次审核"：基于预审建议做出最终裁决（审核通过／退回修改）；
3.预审结论仅为建议、不可自动生效，一切以管理员的二次审核决策为准 |
| 操作完成方向（预设） | 1.进入即默认开始：用户进入即默认开始 接入审核，无需先发起；
2.任务轨道：沿固定任务轨道 N0→N5 逐个（或批量采纳预审建议）完成 待审核队列的二次审核 事项；
3.完成标志：队列中每个智能体均产生一条明确的二次审核决策（审核通过／退回修改）并生效通知，或管理员主动暂停；
4.收尾后仅开放动作：① 查看本次审核批次汇总；② 跟进待办清单；③ 结束场景 |
| 进入方式 | 1.意图路由：命中「审核智能体／有哪些待审核／开始审核」等口令；
2.快捷入口：对话页常驻【智能体审核】（仅管理员可见）；
3.从 待审核队列 页进入。
无论何种方式，进入即由医小管自动打招呼、主动发起 |
| 完成标志 | 1.待审核队列中每个智能体均产生一条明确的二次审核决策（审核通过／退回修改）并生效通知；
2.或管理员主动暂停（未决项保持待审核）；
3.场景锁释放（需可被系统判定） |
| 产出物 | 1.每个已审智能体的二次审核记录（审核结论、待修改说明、审核人、审核时间）；
2.状态流转（审核通过→已纳管／退回修改→退回申请人）；
3.对申请人的通知；
4.本次审核批次汇总 |
| 预计步数 / 时长 | 约 5–6 步、每条 2–3 分钟 |

## 二、任务轨道（状态机节点表）

> 每节点独立一行；`动作`只写要点，通用规则不写；`出口判据`是节点被视为已完成的系统条件；`工具`只列本节点会触发的工具调用。
> 

| **节点** | **名称** | **动作（要点）** | **完成条件** |
| --- | --- | --- | --- |
| N0 | 场景进入·自动打招呼 | 1.固定开场：「你好！我是医小管。目前待审核 X 个智能体，我已完成预审并给出结论」（X 来自 list_pending）；
2.逐条呈现待审队列：智能体编号、名称、预审结论（建议通过／建议退回修改）、具体说明摘要 | 医小管已自动打招呼、待审队列消息已呈现 |
| N1 | 候选定位 | 1.说明可用方式并等待管理员响应；
2.支持批量通过：仅可批量采纳预审"建议通过"项；
3.支持按编号/名称/自然语言指代（如"那个儿科的""第2个"）定位单个；
4.支持按预审结论筛选（如"只看要退回的"）；
5.医小管解析后回述锁定对象或范围 | 据管理员响应锁定审核对象（批量集合或单个） |
| N2 | 专项查看（退回修改必经） | 1.呈报注册信息摘要（名称、版本、所属科室、诊疗环节、功能描述、来源、供应商、技术联系人、联系方式、接入方式及分支字段，API key 密文）；
2.呈报预审五维核验结果（信息完整性、备案材料齐备性、联通测试、名称查重、合规性），逐项标注问题项；
3.支持"看完整详情"在对话内展开注册信息、备案材料、联通测试报告 | 专项查看内容呈现完成（退回修改必须先经本节点） |
| N3 | 二次审核决策 | 1.请管理员给出结论，取值仅两种：审核通过／退回修改；
2.退回修改必须含待修改说明（指明问题项／需整改内容），缺失则追问补齐；
3.若与预审结论不一致，播报差异并请二次确认，记录"覆盖预审"标记；
4.医小管不替代裁决、不诱导结论 | 管理员给出结论＋（退回修改时）待修改说明，并完成二次确认 |
| N4 | 决策生效与通知 | 1.调用 submit_review 落库审核记录（审核结论、待修改说明、审核人＝当前管理员、审核时间 YYYY-MM-DD HH:MM）；
2.按结论流转状态：审核通过→智能体正式纳管；退回修改→退回申请人并附待修改说明；
3.触发对申请人的通知；
4.以对话消息回执处理结果（单条或批量），随后回到 N1（队列计数相应减少） | 该智能体审核记录落库、状态流转、通知已发出 |
| N5 | 批次收尾汇总 | 1.队列清空或管理员"暂停/结束"时触发；
2.播报审核批次汇总：本次审核通过 X 个、退回修改 Y 个、未处理 Z 个（暂停时保持待审核）；
3.固定收尾语，场景结束，不再执行其他动作 | 汇总播报完成、场景锁释放 |

## 三、字段清单

> 只列本场景相关字段；类型 / 校验优先复用《平台通用规范 v1》第九章基线；差异在「规则」列写明。
> 

| **字段** | **来源** | **类型** | **规则（差异化）** |
| --- | --- | --- | --- |
| 预审结论 | 系统预审生成 | 单选枚举 | 1.取值：建议通过／建议退回修改；
2.五维均达标→建议通过；任一维度存在问题→建议退回修改；
3.只读参考，不作为生效依据 |
| 预审说明 | 系统预审生成 | 长文本 | 1.由五维核验依据摘要构成（信息完整性、备案材料齐备性、联通测试、名称查重、合规性）；
2.支撑预审结论，逐项可回溯 |
| 审核结论 | 管理员决策 | 单选枚举 | 1.取值仅两种：审核通过／退回修改；
2.唯一生效依据，医小管不得代管理员做出；
3.仅预审"建议通过"项可批量审核通过，且须管理员确认；
4.退回修改不可批量，须逐个决策 |
| 待修改说明 | 管理员填写 | 长文本 | 1.退回修改必填，需指明问题项／需整改内容；
2.审核通过无需填写；
3.支持手动／语音／文字三种等效输入 |
| 是否覆盖预审 | 系统标记 | 布尔 | 1.管理员决策与预审结论不一致时自动置"是"；
2.必须经二次确认后落库；
3.供审计追溯 |
| 审核人 | 系统自动记录 | 短文本 | 1.取当前登录管理员；
2.不可修改 |
| 审核时间 | 系统自动记录 | 日期时间 | 1.决策生效时间；
2.格式 YYYY-MM-DD HH:MM |
| 被审核智能体信息（引用） | 接入注册场景采集 | 结构化引用 | 1.沿用接入注册场景字段口径：名称、版本、所属科室、诊疗环节、功能描述、来源、供应商、技术联系人、联系方式、接入方式（API／SDK／OTel）及分支字段；
2.API key、平台密钥全程密文（********）呈现；
3.申请人敏感信息按权限最小化呈现，不明文复述、不外泄 |

特殊情况说明：

1. 退回修改必须先经"专项查看"（N2）后方可决策，不得跳过；医小管应固定拒绝"跳过专项查看直接退回"的越界请求。
2. 管理员决策覆盖预审建议时，必须播报差异并请二次确认，落库后置"是否覆盖预审＝是"以留痕。
3. 同一智能体已被其他管理员正在审核时，医小管发送"占用提示消息"提示占用，避免并发重复审核；同一智能体不重复出结论。

## 四、 完整示例对话

> 至少 1 条走通全轨道的对话；建议包含 1 次跑偏 + 1 次校验失败 + 1 次「修改××」。
>
