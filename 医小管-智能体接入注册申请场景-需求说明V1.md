# 医小管-智能体接入注册申请场景-需求说明

## 一、场景定位

| **项目** | **说明** |
| --- | --- |
| 场景名称 | 接入申请 |
| 平台智能体 | 医小管（统一对话页的平台智能体，用户进入场景即自动打招呼、主动发起） |
| 操作完成方向 | 1. 用户进入场景即默认开始（无需用户先发起交互），医小管自动打招呼并邀请上传材料或文字/语音描述；
2. 用户提供任意格式材料/描述，医小管智能识别并自动填充注册表单；
3. 就缺失字段逐项追问、用户回复补全，直至所有必填字段填写完成；
4. 技术信息填写完成后自动发起联通测试；
5. 核验备案材料是否缺失，对缺失的备案材料依据已填写字段自动生成 PDF 版材料；
6. 全部字段与材料齐备后由用户确认；
7. 确认无误后提交注册申请，系统按"科室编号-准入顺序号"自动生成智能体编号；
8. 提交播报后不再执行任何其他动作，场景结束。 |
| 交互方式 | 1. 自动发起：用户进入本场景即默认开始接入注册，无需用户先发起交互——医小管自动打招呼并邀请发材料/语音/文字；
2. 多模态免手动：全过程支持三种等效方式——手动操作（点击卡片按钮/在卡片编辑/上传文件）、语音交互（口述/语音描述）、文字交互（打字）；
3. 三者完全等效，用户无需手动操作也能完成（上传文件属提供材料，非界面操作）。 |
| 进入方式 | 1. 意图路由：用户说"我要接入智能体 / 我要做纳管申请 / 帮我注册一个智能体"等命中进入；
2. 快捷入口：对话页常驻【智能体接入注册】；
3. 无论何种进入方式，进入即由医小管自动打招呼、主动邀请，无需用户先开口。 |
| 输入与留存约定 | 1. 输入接受任意格式：产品说明书/技术规格书等文件（PDF、DOC、DOCX、XLSX、csv、jpg、jpeg、png、链接等），以及文字、语音描述；
2. 单文件不超过 30M，支持多文件；
3. 正式备案材料以 PDF 留存：用户已提供合规 PDF 则直接采用，否则由系统依据已确认字段自动生成 PDF 版备案材料。 |
| 完成标志 | 1. 全部必填字段校验通过并经用户确认；
2. 备案材料齐备（上传或自动生成）；
3. 联通测试完成；
4. 用户确认无误；
5. 注册申请提交成功、智能体编号生成并播报，场景锁释放。 |
| 产出物 | 1. 一条完整的智能体接入注册申请记录（含基本信息、技术信息、备案材料）；
2. PDF 版备案材料（上传或自动生成）；
3. 联通测试报告；
4. 自动生成的智能体编号（科室编号-准入顺序号，如 0001）。 |
| 预计步数 / 时长 | 约 8 步（N0–N7） |

## 二、任务轨道

> **场景类型**：顺序表单型任务——入场后按固定顺序推进，用户无需主动选择路径（区别于"意图路由型"）。
**主流程**：自动开场 → 多模态识别与自动填充 → 缺失字段对话补全 → 自动联通测试 → 备案材料核验（缺失时自动生成 PDF） → 汇总确认 → 提交并生成智能体编号 → 场景结束。
> 

### 1. 开启对话入口

入口方式：

**方式一：首页对话框直接发送消息（文字/语音），以意图命中自然进入接入申请；**

用户说"我要接入智能体 / 我要做纳管申请 / 帮我注册一个智能体"等命中口令，直接进入场景，由医小管自动打招呼并邀请提供材料。

**方式二：首页点击业务标签「接入申请」快捷开启对话；**

如果是通过方式二点击业务标签「接入申请」发起对话，由医小管主动发送固定开场语：

「你好！我是医小管。把产品说明书 / 技术规格书发给我（支持 PDF、DOC、DOCX、XLSX、csv、jpg、jpeg、png、链接等任意文件格式），或文字、语音描述，我来帮你进行智能体接入信息注册申请～」

**两种方式均在左侧「历史会话」新建并进入对应的对话窗口，后续多轮对话在同一窗口内持续进行。无论何种进入方式，进入即由医小管自动打招呼、主动邀请，无需用户先开口。**

### 2. 意图识别

医小管对用户每轮输入先做意图判定，归入以下类别之一，并分流至对应处理：

**意图一：提供材料 / 补全字段（对应 3 执行对应意图 · 意图一）**

- 判定依据：用户上传任意格式材料（单文件≤30M，支持多文件），或发送文字/语音描述智能体接入信息，或对追问项作答；
- 后续处理：进入材料与描述采集、智能识别与自动填充、缺失字段对话补全流程。

**意图二：确认提交 / 修改字段（对应 3 执行对应意图 · 意图二）**

- 判定依据：字段与材料齐备后，用户以点击、语音、文字任一方式发出「确认提交」意图；或在任一节点用户（点/说/打字）"修改××"；
- 后续处理：修改类先定位目标字段/节点处理后回到中断处；确认类进入联通测试 → 备案材料核验 → 汇总确认 → 提交注册申请流程；若技术信息被修改，则自动重跑联通测试并据此刷新自动生成的备案材料。

### 3. 执行对应意图

根据 2 意图识别的结果，分流至以下两条执行链路；分支内步骤按 MECE 原则互斥、依序推进。

**3.1 意图一 ：获取注册信息**

> 通过「多模态材料识别与自动填充 → 缺失字段对话补全」两步，采集并校验全部必填字段。
> 

**A. 多模态材料识别与自动填充**

1. 接收用户上传的任意格式材料（PDF、DOC、DOCX、XLSX、csv、jpg、jpeg、png、链接等，单文件≤30M，支持多文件）、文字或语音，并对每个输入做基础校验（大小、可解析性），不可解析时提示改用其他格式或改为文字描述；
2. 对材料 / 描述执行多模态识别（OCR / 文档解析 / ASR 语音转写 / 链接抓取）；
3. 自动填充注册表单草稿（智能体名称、版本、功能描述、所属科室线索、诊疗环节线索、智能体来源与供应商、技术联系人、联系方式、接入方式、接口地址、API key 等）；
4. 播报已填充字段与仍缺失字段清单。

**B. 缺失字段对话补全**

1. 针对缺失或需确认的字段逐项对话追问（每轮一问），用户以手动 / 语音 / 文字任一方式回答；
2. 逐项采集与校验：
    - 智能体名称：2–20 字符，名称查重，重名提示"此名称已被使用，请重新命名"；
    - 版本：数字.数字；
    - 所属科室：科室字典（代码＋名称）；
    - 诊疗环节：9 选项，选"其他"填空 ≤20 字；
    - 功能描述：四要素，≤500 字，确认卡确认或修改；
    - 智能体来源：自研 / 第三方 / 合作研发；
    - 供应商名称：≤30 字；
    - 技术联系人：2–10 字；
    - 联系方式：11 位手机号，失败提示"请输入正确的11位手机号"；
    - 接入方式：API / SDK / OTel 及分支字段；
3. 循环直至所有必填字段校验通过；
4. 智能体编号不询问（由系统在提交时按"科室编号-准入顺序号"自动生成）。

**3.2 意图二 ：确认提交与提交注册申请**

> 依次进入自动联通测试 → 备案材料完整性核验 → 汇总确认与提交三步；期间若用户对任一字段"修改××"，定位目标节点处理后回到中断处，技术信息被修改时自动重跑联通测试并据此刷新自动生成的备案材料。
> 

**A. 自动联通测试**

1. 技术信息填写完成后自动发起联通测试；
2. 按接入方式分支执行：
    - API 接入：用接口地址＋API key 调用连通性探测；
    - SDK 接入：校验平台 URL 地址、平台密钥 key 与埋点代码可用性；
    - OTel 接入：校验平台 URL 地址可达性；
3. 返回测试结果（成功 / 失败 ＋ 耗时 / 错误信息）；
4. 失败则定位到对应技术字段引导修正后自动重测。

**B. 备案材料完整性核验**

1. 核验正式备案材料是否齐备：
    - 产品说明书：须含产品名称、产品简介、主要功能、开发单位及技术联系人、产品版本等；
    - 技术规格书：须含接口地址、认证方式、请求参数、返回参数、数据格式、请求 / 返回示例、错误码说明等；
2. 用户已上传合规 PDF 且内容达标的直接采用；缺失或非 PDF 的根据已填写字段自动生成 PDF 版材料；
3. 发送材料生成卡供用户预览确认（可下载、可要求调整）；
4. 备案材料遵循"先核验、后生成"，生成动作安排在联通测试之后——因技术规格书含接口地址、认证方式等技术字段，联通测试可能触发技术字段修正，将生成动作后置可保证自动生成的 PDF 材料与最终确认字段完全一致。

**C. 汇总确认与提交**

1. 以对话消息逐项呈现：基本信息 ＋ 技术信息 ＋ 备案材料清单 ＋ 联通测试结果（API key 密文）；
2. 每项可【修改】；用户点 / 说 / 打字"修改××" → 跳转对应节点处理后返回本节点重新汇总；
3. 确认提交同样支持点 / 说 / 打字，三种输入方式等效；确认后提交注册申请；
4. 系统按"科室编号-准入顺序号（如 0001）"自动生成智能体编号，回执提交成功（智能体编号、名称、版本），并以固定收尾语致谢；
5. 提交播报后不再执行任何其他动作。

### 4. 对话延续及收尾

1. 每一步的确认 / 选择 / 修改 / 补填均支持手动操作、语音、文字三种等效方式；
2. 单步完成后医小管保持接收状态，等待用户继续操作（补填、修改、确认提交等）；
3. 注册申请提交成功、智能体编号生成并播报后，本场景自动结束，不再执行任何其他动作，场景锁释放。

### 5. 特殊情况说明

**特殊情况1：跑偏拉回**

情况定义：任一节点用户询问内容跑偏（含请求其他平台操作），偏离当前接入注册申请。

处理方式：

1. 发送固定话术（不可改写）："我们先来完成智能体接入注册申请，稍后再为您解决此问题"，随后复述当前问题；
2. 不切换场景、不对无关话题展开解答。

**特殊情况2：用户放弃**

处理方式：

1. 用户主动放弃后，保存草稿并释放场景锁；
2. 草稿（含已上传材料与已填字段）保留 7 天。

**特殊情况3：校验 / 测试失败**

情况定义：涉及自动联通测试失败与字段格式 / 查重校验失败两类，处理逻辑统一为"定位失败对象 → 引导修正 → 重试"。

处理方式：

1. 联通测试失败：
    - 首次失败时定位到对应技术字段引导修正后自动重测，重测通过方视为完成；
    - 若用户选择不修正，则转为保存草稿后结束场景（同特殊情况2），不视为该节点完成；
2. 字段校验失败：
    - 智能体名称重名 → 固定文案"此名称已被使用，请重新命名"，要求重新命名；
    - 联系方式格式不符 → 固定文案"请输入正确的11位手机号"，要求重新输入；
    - 其他校验失败字段回到"缺失字段对话补全"追问流程，直至通过。

**特殊情况4：敏感字段密文**

1. API key 与平台密钥全程密文，不明文回显、不朗读、不写入回复文本；
2. 仅通过密钥卡 icon 切换显示 / 隐藏或复制。

3. 除以上字段外，禁止创建、询问或记录任何新增字段。

## 四、 完整示例对话

> 示例背景：内分泌科与第三方公司合作建设的"糖尿病随访助手"申请接入平台，接入方式为 API。示例演示：进入即自动发起、材料识别填充、缺失字段以手动/语音/文字补全、重名校验、联通测试失败与修正、缺失材料自动生成 PDF、汇总与提交全流程。
> 

| **节点** | **角色** | **对话内容** | **系统动作 / 组件** |
| --- | --- | --- | --- |
