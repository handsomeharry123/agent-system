# 医疗智能体监控指标体系V1.1

<aside>
💡

本指标体系基于[医疗智能体统一管理运营平台功能需求说明](https://www.notion.so/9fcf123c6c7482038c3c8157af89b26e?pvs=21)中「统一运行监控中心」四大维度（性能监控、状态监控、业务监控、成本监控）展开，明确每个维度下的核心监控指标、计算口径、采集方式、告警阈值与责任主体，为医疗智能体上线后的稳定运行与持续优化提供量化依据。

</aside>

## 一、指标体系总览

本章作为全篇索引，汇总性能、状态、业务、成本四大监控维度的指标分组与数量，便于快速查阅与定位；具体定义、计算口径、单位与阈值见 第二章 ~ 第五章。

| **监控维度** | **监控目标** | **核心指标分组** | **指标数量** |
| --- | --- | --- | --- |
| 性能监控 | 保障智能体的响应效率与服务能力 | 响应时延、并发吞吐、准确率、稳定性、智能体行为 | 18 |
| 状态监控 | 实时掌握运行健康度并联动台账 | 运行状态、健康检查、异常告警、依赖状态 | 10 |
| 业务监控 | 衡量智能体的业务价值与服务质量 | 调用量、任务完成、内容合规细分、用户反馈、协同效能 | 15 |
| 成本监控 | 按日/周/月/年统计总成本与各维度占比，保障整体成本可控 | 资源消耗、资源利用、消耗汇总与趋势 | 16 |

## 二、性能监控指标

衡量智能体在响应效率、并发承载、回答准确度、运行稳定性与智能体行为等方面的表现，是评估上线智能体能否满足临床业务节奏的核心维度；其中"智能体行为类"为 5.29 演示重点要求补充的循环、工具调用监控。

### 2.1 响应时延类

刻画请求级响应速度，覆盖平均与尾部时延、流式首 Token 等关键体感维度。

| **指标名称** | **定义与计算口径** | **单位** | **建议阈值** | **可视化方式** |
| --- | --- | --- | --- | --- |
| 平均响应时延 | 统计周期内，智能体从接收请求到返回首个完整结果的平均耗时 | ms | ≤ 2000 ms | 折线图（x=时间 5min/1h 粒度，y=ms）；2000ms 红色虚线阈值，超阈区段填充浅红 |
| P95 响应时延 | 95% 的请求响应时延不超过该值 | ms | ≤ 5000 ms | 多线折线图（P50/P95/P99 三线同图，蓝/橙/红色区分）；5000ms 红色虚线阈值 |
| P99 响应时延 | 99% 的请求响应时延不超过该值 | ms | ≤ 8000 ms | 与 P95 同图叠加（见上行）；8000ms 红色虚线阈值 |
| 首 Token 时延（TTFT） | 流式输出场景下，从请求到首个 Token 返回的耗时 | ms | ≤ 800 ms | 左右双视图：左折线图（均值随时间），右直方图（耗时分布）；800ms 红色虚线 |

## 三、状态监控指标

## 五、成本监控指标

围绕算力、存储、流量、模型四类资源，**以消耗量为主口径**采集与监控，单价确认后再换算金额，支撑成本可控与资源高效利用。

<aside>
💰

**计量口径说明**：医院侧底座模型、云厂商、存储/网络计费规则尚未确认时，本维度**先按消耗量（核·h、GB·月、GB、tokens、次）采集与告警**；金额类指标（元、元/次、元/1k tokens）作为可配置项保留，待医院方确认单价及计费方式后，由平台侧 `金额 = 消耗量 × 单价` 自动换算启用。这也是 AWS Cost Explorer、Datadog、Langfuse 等主流产品的通用做法——用量与金额始终是两条独立的数据轴。

</aside>

### 5.1 资源消耗类

按算力 / 存储 / 流量 / 模型四类资源记录原生消耗量，是成本视图的底层数据来源，不依赖单价即可立即采集。

| **资源类型** | **指标名称** | **定义与计算口径** | **单位** | **建议阈值** | **可视化方式** |
| --- | --- | --- | --- | --- | --- |
| 算力 | CPU 使用核时 | 统计周期内所有实例 CPU 累计占用核时 | vCPU·h | 按预算约束 | 堆叠面积图（x=时间 y=vCPU·h），按智能体堆叠多色；可切换为按科室堆叠 |
| 算力 | GPU 使用卡时 | 统计周期内所有实例 GPU 累计占用卡时 | 卡·h | 按预算约束 | 堆叠面积图（按智能体或卡型号堆叠）+ 旁边 TOP5 智能体横向条形图 |
| 存储 | 块/云盘存储用量 | 实例本地盘、云盘平均占用容量 | GB·月 | 按预算约束 | 面积折线图（单调累积），按存储类型分层（系统盘/数据盘/快照） |
| 存储 | 对象/向量库存储用量 | 对象存储（文档、日志、模型权重）+ 向量索引平均占用容量 | GB·月 | 按预算约束 | 面积折线图（按对象存储/向量索引/模型权重三层堆叠） |

## 六、采集、分级与告警机制

### 6.1 数据采集方式

- **埋点上报**：智能体 SDK / 网关在请求入口与出口埋点，采集时延、错误码、Token 消耗、用户反馈等运行时数据。
- **平台主动探测**：监控中心通过心跳、健康检查接口与依赖探活，采集运行状态与依赖可用率。
- **资源观测**：对接底层算力、存储、网络观测系统，采集资源使用与成本相关数据。
- **业务回流**：从 HIS、电子病历、工单系统回流任务完成、人工修改、投诉等业务结果数据。

### 6.2 指标分级

| **等级** | **说明** | **典型指标** |
| --- | --- | --- |
| P0 核心 | 直接影响业务可用性与患者安全，必须实时监控并强告警 | 服务可用率、回答准确率、内容合规率、运行状态 |
| P1 重要 | 影响用户体验与业务效率，分钟级监控并告警 | P95 时延、错误率、任务完成率、工具调用失败率、循环超限率 |
| P2 一般 | 用于持续优化与成本分析，按小时/天级汇总 | 实例闲置率、单次调用成本、调用量、活跃用户数 |

### 6.3 告警与处置流程

1. **阈值触发**：监控中心按等级与阈值实时判定，自动生成告警事件并写入台账。
2. **分级通知**：P0 电话+短信+IM 同步通知运维与业务负责人；P1 IM+邮件通知；P2 日报汇总。
3. **自动处置**：
    - **内容实时阻断**：P0 级内容安全异常（不合规、过度承诺、歧视性表达、不当语气等）在输出网关侧实时拦截，替换为安全话术或转人工；P1 级仅打标+告警放行（吴老师明确要求的产品差异化能力，原话："假设监测到输出的时候，用到了一些不合理的这种输出某些词，我能够阻断吗？"）。
    - **资源类异常**：自动扩缩容、自动隔离异常实例、自动切换备份模型。
    - **异常数据回流**：P0/P1 异常事件对应的会话原文自动打标，推送至「数据资产」模块作为后续微调训练样本，形成数据飞轮闭环。
4. **复盘归档**：异常事件闭环后写入安全治理与运营复盘记录，反哺迭代优化建议。

## 七、诊断与评测联动能力

<aside>
🔗

🚧 **本章规划为二期能力**。吴老师在 5.29 演示中强调两点：① "**在这个监测过程中，可能是可以诊断**"——诊断不是独立模块，而是嵌入在监控过程中的能力；② "**还能通过评测，能够对技术整体开展评价**"——评测既是接入准入门槛，也是持续性的技术评价。一期上线优先完成 §二 ~ §六 指标采集与告警闭环；自动根因定位、诊断报告生成、运行期回归评测、退场建议等能力作为二期专项落地，本章列出的指标与阈值供产品规划参考，**不纳入一期 SLA 与验收范围**。

</aside>

### 7.1 嵌入式诊断指标

| **指标名称** | **定义与计算口径** | **单位** | **建议阈值** | **可视化方式** |
| --- | --- | --- | --- | --- |
| 根因定位命中率 | 异常事件中能自动定位到根因（模型/工具/数据/网络）的占比 | % | ≥ 80% | 折线图（80% 下限红线）+ 根因分类饼图（模型/工具/数据/网络 四色） |
| 诊断报告生成率 | P0/P1 异常事件中自动生成诊断报告的占比 | % | ≥ 95% | 折线图（95% 下限红线）+ 未生成报告事件清单（异常事件 ID 列表） |
| 诊断闭环时长 | 从异常发生到处置完成的平均时长 | min | P0 ≤ 15 / P1 ≤ 60 | 分组直方图（P0/P1 两组并排，x=耗时分桶 y=事件数）+ 均值标线 + 阈值红线 |

### 7.2 评测分数联动

- **准入评测分数**：能力 / 安全 / 伦理 / 鲁棒性 四维分数作为单智能体看板的固定模块，与日常运行指标对照展示。
- **运行期评测分数**：基于采集到的真实人机交互数据周期性回归评测，与运行期监控指标形成对照，识别能力衰减。
- **退场建议**：综合运行期分数与监控异常事件，对持续不达标的智能体提出下线或迭代建议，反哺接入流程。

### 7.3 优化建议联动

<aside>
💡

对齐[医疗智能体统一管理运营平台功能需求说明](https://www.notion.so/9fcf123c6c7482038c3c8157af89b26e?pvs=21)中“**基于全过程监控数据与用户反馈，平台梳理生成智能体优化建议**”的闭环要求。优化建议不是独立指标，而是基于本指标体系产出的结果。

</aside>

- **指标趋势驱动**：对持续超阈值的指标（如幻觉率、不合规回答率、循环超限率、GPU 利用率偏低等）自动生成**优化建议卡片**（提示可能原因 + 推荐动作）。
- **闭环闭环型建议**：结合 §7.1 诊断结果与 §7.2 评测分数，输出“调参 / 换模 / 微调 / 下线”四类建议，联动接入中心与数据资产。
- **迭代闭环跨期跟踪**：建议执行后回源对照指标变化，验证优化效果，形成“监控→诊断→评测→建议→迭代”闭环。

- **成本与资源看板**（面向运营与财务）：按智能体、科室、资源类型多维下钻，支持日/周/月/年成本趋势与超支提醒。
